# Phase 迭代规则（强制）

本文档定义 **进入任一 Phase 之前** 与 **Phase 执行过程中** 必须遵守的流程，适用于 `PHASE00`…`PHASE12` 及后续增补。与 **`claude-code-sourcemap`** 还原树的对照为硬要求。

**存放位置（git）**：本文件位于模块根目录，**纳入版本控制**。`docs/` 目录按 `.gitignore` 不提交；若本地在 `docs/phases/` 维护同名草稿，须定期与本文 **对齐**。

**还原树根路径（约定）**：**`claude-code-sourcemap/restored-src/src/`**（文档中常简称 **`src/`**）。Go 实现须与还原语义对齐；路径级对照落在各 Phase 的 **`PHASEXX_SPEC_AND_ACCEPTANCE.md`** **§4**（或等价章节）。

---

## 一、Phase 迭代前（开工门槛）

在进入本 Phase 的代码迭代之前，**须全部完成**下列三项；未完成则不得开始合入本 Phase 功能代码（仅允许补 SPEC / 对照表 / 记录文档）。

### 1. 全量核对功能清单与验收标准

- 通读 **`PHASEXX_SPEC_AND_ACCEPTANCE.md`** 的 **§2 功能清单**、**§3 验收标准**，以及 **`PHASEXX_E2E_ACCEPTANCE.md`**（及该 Phase 若存在的 **`PHASEXX_UI_ACCEPTANCE.md`**）。文档路径：通常为 `docs/phases/`（本地）或与主计划一致的副本。
- 与 **`claude-code-sourcemap`** 还原行为 **全量对齐**：逐项确认「做什么、做到什么程度、如何验收」，不得在未改 SPEC 的情况下擅自缩小范围。
- 若主计划或 **`PARITY_CHECKLIST`**、**`ARCHITECTURE_BOUNDARIES.md`** 对平台豁免有规定，须在 SPEC 或对照表中 **显式引用**，不得口头约定。

### 2. 与 claude-code-sourcemap 的全量路径对照列表

- 本 Phase SPEC 中 **必须** 包含 **还原路径 ↔ Go 交付物** 的对照（推荐独立为 **§4**，与 `PHASE04_SPEC_AND_ACCEPTANCE.md` 等结构一致）。
- 对照须覆盖本 Phase 承诺的 **`src/`** 下文件/目录（含 `services/`、`utils/`、`constants/`、`entrypoints/` 等），并标明 **已实现 / 部分 / 未实现** 或等价状态。
- 大型横切能力（如 `utils/http.ts`）须在对照表中写明 **归属 Phase** 与 **Go 落点包路径**，避免重复实现或遗漏。

### 3. 实现进度（迭代记录）

- 本 Phase SPEC 中 **必须** 设有 **实现进度 / 迭代记录** 章节（推荐 **§6**）。
- 每次合并进主线的实质进展，应 **追加** 记录：日期、批次、完成项、关联 commit 摘要、仍缺项与依赖（如「待 Phase 5」）。
- 迭代记录与 **§2、§3 状态列**（完成/部分）须 **同步修订**，避免文档与代码长期不一致。

## 二、Phase 迭代中（执行与提交）

### 1. 执行计划列表 → 逐项自动执行

- **每次迭代开始前**（或每个工作会话开始时），先 **整理并输出一份可执行计划列表**（条目应小到可独立测试、可独立提交）。
- 按计划 **顺序执行**：实现 → 自测 → 对照 SPEC/E2E 相关条目勾选或更新状态。
- 不得跳过计划中的验收步骤；若发现 SPEC 有误，应先改 SPEC 再改代码，或记录明确偏差理由。

### 2. 测试验收后自动 commit，直至本 Phase 完成

- **每完成计划中的独立条目**（或逻辑上可独立 review 的一组变更），在 **通过约定测试**（如本 Phase 的 `make test-phaseX` 或 SPEC 写明之命令）后，**立即 `git commit`**（信息应能对应计划条目）。
- **持续循环**：commit → 下一条目 → …，直至 **§2/§3 与本 Phase E2E** 均满足「本 Phase 完成」定义。
- Phase 收尾时：**§6 迭代记录** 写明清仓结论；必要时更新 **§4 路径对照** 最终状态。
